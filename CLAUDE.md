# CLAUDE.md

이 파일은 이 프로젝트(알람 앱) 전체에서 항상 지켜야 할 공통 규칙을 담는다.
특정 기능(예: 뉴스)만의 세부 규칙은 여기 두지 않고 해당 폴더의 `CLAUDE.md`에 별도로 둔다.

## 🚨 최우선 보안 규칙: API 키를 프론트엔드에 절대 넣지 않는다

- **네이버 API 키(Client ID/Secret)를 포함해, 외부 서비스의 API 키·시크릿은 어떤 경우에도 프론트엔드(브라우저에서 실행/로딩되는) 코드에 넣지 않는다.**
- API 키는 서버 환경변수로만 관리하고, 서버 라우트/서버리스 함수 핸들러 내부에서만 읽는다.
- `NEXT_PUBLIC_` 접두사 등 클라이언트에 노출되는 방식으로 키를 다루지 않는다.
- 프론트엔드 번들, 네트워크 응답(브라우저가 받는 응답), 콘솔 로그 어디에도 키 값이 노출되어서는 안 된다.
- 브라우저는 외부 API를 직접 호출하지 않고, 자체 서버 엔드포인트만 호출한다. 외부 API 호출과 키 사용은 항상 서버 쪽 코드에 격리한다.
- 새로운 외부 연동을 추가할 때도 이 원칙을 동일하게 적용한다.

## 실패 허용 원칙: 알람 소리는 반드시 울려야 한다

- **부가 기능(뉴스 브리핑 등)이 실패해도 알람 소리 자체는 반드시 울려야 한다.**
- 부가 기능의 조회 실패, 렌더링 오류, 예외 등은 알람 트리거·소리 재생 로직과 완전히 분리되어야 하며, 그 예외가 알람 타이머/체크 루프까지 전파되어 알람이 울리지 않는 상황이 발생해서는 안 된다.
- 알람의 핵심 동작(등록된 시각에 소리를 울리는 것)은 다른 어떤 기능보다 우선하는 신뢰성 요구사항이다. 새 기능을 추가할 때 이 핵심 흐름에 영향을 줄 수 있는 코드 경로가 없는지 항상 확인한다.

## 데이터 저장 원칙

- 별도 요구가 없는 한, 외부 API로 조회한 데이터는 DB/파일/인메모리 캐시에 영속 저장하지 않고 매 요청마다 실시간으로 조회한다.
- 외부 API 호출 시 캐시를 사용하지 않는다(`cache: 'no-store'` 등).

## 작업 순서 원칙

새 기능을 구현할 때는 다음 순서를 지킨다.

1. 먼저 화면(UI)을 가짜 데이터(mock data)로 만들어 동작을 확인한다.
2. 그다음 실제 백엔드/서버 라우트를 연결한다.
3. 예외 처리(오류 상태 표시, 실패 시 폴백 등)는 가장 마지막에 추가한다.

## 미확정 사항을 임의로 결정하지 않는다

- 각 기능의 PRD에 있는 "Open Questions"(예: 뉴스 기능의 검색 키워드 방식, 재생 타이밍 등)는 아직 확정되지 않은 사항이다.
- 이런 미확정 사항을 코드 작성 중 임의로 추측·결정하지 않는다. 확정이 필요하면 작업을 진행하기 전에 사용자에게 확인한다.

## 응답 정제 및 렌더링

- 외부에서 받아온 텍스트(HTML 태그, HTML 엔티티 등이 섞여 있을 수 있는 데이터)는 서버 쪽에서 정제한 뒤 클라이언트에 전달한다.
- 렌더링 시 프레임워크의 기본 텍스트 이스케이프에 의존해 XSS 위험을 낮춘다.
- 외부 링크는 `target="_blank" rel="noopener noreferrer"`로 새 탭에서 연다.
