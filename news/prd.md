# 최근 뉴스 조회 및 음성 읽기 기능 — PRD

> 근거 문서: `최근뉴스_기술명세서.md` (기술 명세). 본 PRD는 해당 기술 명세를 바탕으로 제품 요구사항 관점에서 재정리한 문서다.

## 1. 배경 및 목표

### 배경

알람 앱은 현재 시계·날씨·알람 등록/알림 기능만 제공한다. 사용자가 아침에 알람을 확인하거나 앱을 여는 시점에, 특정 주제에 한정하지 않은 최신 뉴스를 별도 검색 없이 바로 확인할 수 있도록 "최근 뉴스" 패널을 추가한다.

뉴스는 네이버 검색 API를 통해 서버(라우트 핸들러)에서만 조회하며, 브라우저는 자체 서버 엔드포인트(`/api/news`)만 호출한다. 조회된 뉴스는 목록으로 표시되고, 사용자가 원하면 브라우저 내장 음성 합성(Web Speech API)으로 읽어준다.

### 목표

- 사용자가 별도 조작 없이 앱 진입 시 최신 뉴스 5건을 자동으로 확인할 수 있다.
- 원하는 뉴스를 개별 또는 전체 음성으로 들을 수 있다.
- 네이버 API 키가 브라우저에 노출되지 않는 구조를 보장한다.
- 기존 알람(시간 도달 시 소리 재생) 기능의 신뢰성에 어떤 영향도 주지 않는다.

## 2. 사용자 스토리

- 사용자로서, 앱을 열었을 때 최신 뉴스 5건이 자동으로 로딩되어, 별도로 검색하지 않아도 최근 소식을 확인할 수 있다.
- 사용자로서, 특정 뉴스 항목의 ▶ 버튼을 눌러 제목과 요약을 음성으로 들을 수 있다.
- 사용자로서, "전체 듣기"를 눌러 5건의 뉴스를 순서대로 이어서 들을 수 있다.
- 사용자로서, 재생 중인 뉴스를 다시 클릭해 즉시 멈출 수 있다.
- 사용자로서, 뉴스 목록을 최신 상태로 갱신하고 싶을 때 새로고침 버튼으로 다시 불러올 수 있다.
- 사용자로서, 뉴스 조회가 실패하거나 네트워크 문제가 있어도, 등록해 둔 알람은 정해진 시각에 반드시 울린다는 것을 신뢰할 수 있다.
- 사용자로서, 관심 있는 뉴스 원문을 확인하고 싶을 때 링크를 눌러 안전하게 새 탭에서 열 수 있다.

## 3. 기능 요구사항

### 3.1 뉴스 조회

- FR-1. 프론트엔드는 서버 라우트 `GET /api/news`만 호출하며, 네이버 API 엔드포인트나 키를 직접 알지 못한다.
- FR-2. 서버 라우트는 네이버 검색 API(`https://openapi.naver.com/v1/search/news.json`)를 `sort=date`(최신순), `display=5`(고정 5건) 조건으로 호출한다. 검색 대상은 특정 주제(예: 스포츠, 야구 등)에 한정하지 않은 전체 최신 뉴스로 한다.
- FR-3. 서버 라우트는 응답의 `title`, `description`에 포함된 HTML 태그(`<b>` 등)와 HTML 엔티티(`&quot;`, `&amp;`, `&lt;`, `&gt;`, `&nbsp;`, `&#39;` 등)를 정제한 뒤 클라이언트에 전달한다.
- FR-4. 링크는 `originallink`를 우선 사용하고, 없으면 `link`를 사용한다.
- FR-5. 클라이언트 응답 스키마는 `{ items: [{ title, description, link, pubDate }] }`이며, 오류 시 `{ error: string }` 형태로 응답한다.
- FR-6. 뉴스 목록은 캐싱 없이 매 요청마다 실시간으로 조회한다 (관련 세부 기준은 [[비기능 요구사항]] 참조).

### 3.2 화면 동작 (`NewsPanel`)

- FR-7. 앱(알람 화면) 진입 시 `NewsPanel`이 자동으로 뉴스 조회를 1회 수행한다.
- FR-8. 새로고침 버튼 클릭 시 뉴스를 다시 조회한다. 조회 중에는 버튼을 비활성화하고 "불러오는 중…" 상태를 표시한다.
- FR-9. 각 뉴스 항목에는 개별 재생(▶) 버튼이 있으며, 클릭 시 해당 항목의 제목+설명을 음성으로 읽는다. 재생 중 같은 항목을 다시 클릭하면 정지한다.
- FR-10. "전체 듣기" 버튼 클릭 시 5건을 "N번째 뉴스. 제목. 설명." 형식으로 순서대로 이어 읽는다. 재생 중 클릭 시 즉시 정지한다.
- FR-11. 새로운 음성 재생을 시작하면 이전에 재생 중이던 음성은 즉시 취소한 뒤 새로 시작한다 (동시 재생 방지).
- FR-12. 뉴스 목록이 비어 있으면 안내 문구를, 조회 오류 시 오류 메시지를 표시한다.
- FR-13. 뉴스 원문 링크는 `target="_blank" rel="noopener noreferrer"`로 새 탭에서 연다.

### 3.3 음성 읽기

- FR-14. 음성 합성은 Web Speech API(`SpeechSynthesisUtterance`)를 사용하며 서버를 경유하지 않고 브라우저에서 직접 처리한다.
- FR-15. 언어는 `ko-KR`로 고정한다.
- FR-16. `'speechSynthesis' in window` 체크로 미지원 브라우저를 감지하고, 미지원 시 재생 버튼을 비활성화하며 안내 문구를 표시한다.

## 4. 비기능 요구사항 (보안 최우선)

보안 관련 항목을 최우선으로 명시한다. 아래 세 가지는 반드시 지켜야 하는 필수 요구사항이다.

- **NFR-1 (필수). 네이버 API 키(Client ID/Secret)는 프론트엔드에 절대 노출하지 않는다.**
  - `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`은 서버 환경변수로만 관리하고 서버 라우트 핸들러 내부에서만 읽는다.
  - `NEXT_PUBLIC_` 접두사(또는 이에 준하는 클라이언트 노출 방식)를 절대 사용하지 않는다.
  - 프론트엔드 번들, 네트워크 응답, 콘솔 로그 어디에도 키 값이 포함되지 않아야 한다.
- **NFR-2 (필수). 뉴스 데이터는 DB에 저장하지 않고 매 요청마다 실시간 조회한다.**
  - 서버 라우트는 네이버 API 호출 시 캐시를 사용하지 않는다 (`cache: 'no-store'` 등).
  - 별도의 데이터베이스, 파일, 인메모리 캐시에 뉴스 원문이나 정제 결과를 영속 저장하지 않는다.
  - (향후 캐싱을 도입하는 경우는 본 PRD의 범위 밖이며, [[범위 제외]] 참조)
- **NFR-3 (필수). 뉴스 브리핑 기능이 실패해도 알람 소리 자체는 반드시 울려야 한다.**
  - 뉴스 조회 실패, 음성 합성 실패, `NewsPanel` 렌더링 오류 등은 알람 트리거·소리 재생 로직과 완전히 분리되어 있어야 한다.
  - 뉴스 관련 코드에서 발생한 예외가 알람 타이머/체크 루프까지 전파되어 알람이 울리지 않는 상황이 발생해서는 안 된다.
  - 이 요구사항은 코드 리뷰 및 테스트에서 우선적으로 검증되어야 한다 (참고: [[성공 기준]]).

### 기타 비기능 요구사항

- NFR-4. 네이버 API 오류 응답은 상태 코드를 그대로 클라이언트에 전달하고, 네트워크/예외 상황은 500으로 응답한다.
- NFR-5. 응답 정제 단계에서 HTML 태그를 제거해 XSS 위험을 낮추고, 렌더링 시 React의 기본 텍스트 이스케이프에 의존한다.
- NFR-6. 인증/사용자 구분 없이 개인용 단일 사용자 기준으로 동작한다 (다중 사용자 대응은 범위 밖).
- NFR-7. 음성 품질은 사용자 OS/브라우저 내장 TTS 엔진에 의존하며 기기마다 다를 수 있다는 점을 사용자에게 별도 고지하지 않되, 알려진 제약으로 문서화한다.

## 5. 범위 제외 (Out of Scope)

- 뉴스 데이터 캐싱/DB 저장 및 이를 이용한 배치 갱신(예: Supabase + `pg_cron`) — NFR-2와 배치되므로 본 기능에서는 다루지 않는다.
- 사용자 지정 검색어/주제 입력 UI(관심 분야 설정 등) — 특정 주제에 한정하지 않은 전체 최신 뉴스를 고정으로 제공하며, 개인화된 주제 필터링은 다루지 않는다.
- 뉴스 개수 조절 UI (`display` 값 변경) — 5건 고정.
- 오류 자동 재시도(지수 백오프 등) — 수동 새로고침만 지원.
- 네이버 Clova Voice, Google Cloud TTS 등 외부 TTS 연동 — 브라우저 내장 TTS만 사용.
- 알람 발생 시점에 뉴스를 자동으로 읽어주는 "알람-뉴스 자동 연동" — 현재 정의된 흐름은 사용자가 앱 화면에서 수동으로 재생 버튼을 눌러야 하는 방식이며, 알람 발생과 뉴스 음성 재생을 자동으로 묶는 것은 범위 밖이다 (단, NFR-3과의 관계는 [[open-questions]]에서 확인 필요).
- 다중 사용자 인증/개인화된 뉴스 추천.

## 6. Open Questions

- **최신 뉴스 조회 시 실제 검색 쿼리 문자열이 확정되지 않았다.** 특정 주제(야구 등)에 한정하지 않고 전체 최신 뉴스를 제공한다는 방향은 확정되었으나(기술 명세의 `query=프로야구`는 채택하지 않음), 네이버 뉴스 검색 API는 검색어 파라미터가 필수이므로 "전체 최신 뉴스"를 어떤 쿼리 값으로 요청할지(예: 공백, 특정 범용 단어, 또는 검색 API가 아닌 다른 네이버 API 사용 등)는 기술적으로 확정이 필요하다.
- **NFR-3("뉴스 브리핑 실패해도 알람은 울려야 한다")과 실제 기능 흐름의 관계가 불명확하다.** 기술 명세상 뉴스는 사용자가 수동으로 열람/재생하는 패널 기능으로 설명되어 있고, 알람이 울릴 때 자동으로 뉴스를 브리핑하는 흐름은 명시되어 있지 않다. "뉴스 브리핑"이 (a) 알람이 울릴 때 자동으로 뉴스를 음성으로 읽어주는 별도 기능을 의미하는지, 아니면 (b) 현재의 수동 재생 기능 자체를 포괄적으로 지칭하는 것인지 확인이 필요하다. (a)라면 이 자동 연동 흐름에 대한 별도 기능 요구사항 정의가 추가로 필요하다.
- 뉴스 패널의 정확한 배치/노출 위치("우측 패널")가 현재 알람 앱의 반응형 레이아웃(모바일 등)에서 어떻게 표시될지 확정되지 않았다.
- 네이버 API 요청 실패 시 사용자에게 노출할 오류 메시지의 정확한 문구/UX가 확정되지 않았다.
- 네이버 오픈API 키의 발급 주체, 호출량 제한(쿼터), 초과 시 처리 방침이 확정되지 않았다.
- 프로젝트가 Next.js 기반(App Router, `app/api/news/route.ts`)으로 전환된다는 전제가 기술 명세에 있는데, 현재 저장소는 정적 HTML/vanilla JS + Vercel 서버리스 함수(`api/*.js`) 구조다. 이 기능을 어떤 스택 위에 구현할지(Next.js 도입 여부 포함) 확정되지 않았다.

## 7. 성공 기준

- 앱 진입 시 3초 이내(정상 네트워크 기준)에 뉴스 5건이 표시되거나, 실패 시 명확한 오류 문구가 표시된다.
- 개발자 도구(Network/Sources)로 확인했을 때, 브라우저에 전달되는 어떤 응답/번들에도 `NAVER_CLIENT_ID`/`NAVER_CLIENT_SECRET` 값이 포함되지 않는다.
- 서버 라우트가 매 호출마다 네이버 API를 실제로 재호출함을 확인할 수 있고(캐시 히트 없음), 별도 DB/파일에 뉴스가 저장되지 않는다.
- **의도적으로 뉴스 API를 실패시키는 테스트(네트워크 차단, 잘못된 키 등)를 수행했을 때, 기존에 등록된 알람이 예정된 시각에 정상적으로 소리를 울린다.** 이 시나리오는 배포 전 필수 회귀 테스트 항목으로 포함한다.
- 뉴스 항목 재생/전체 듣기/정지 동작이 의도한 대로 동작하며, 재생 중 다른 재생을 시작해도 두 음성이 겹쳐 들리지 않는다.
- 음성 미지원 브라우저에서 재생 버튼이 비활성화되고 안내 문구가 노출된다.
